# Streaming Systems — Deep Dive 

---

# 1. The Origin Story: Why Streaming Even Exists

Before streaming, the world ran on **batch processing**.

Imagine a company like Flipkart in early days:

* Orders collected all day
* Processing happens at night (batch job)
* Reports ready next morning

### Sounds fine… until it's not.

Now imagine:

* Fraud transaction happens at 10:01 AM
* You detect it at **next day 9 AM**

👉 Money is already gone.

---

### This was the core problem:

> Systems were **observing the past**, not reacting to the present.

Batch systems answered:

* “What happened yesterday?”

But businesses needed:

* “What is happening *right now*?”

---

# 2. The Evolution Path (Very Important Story)

### Step 1: Batch Systems (Hadoop Era)

* High throughput
* Very high latency (hours)
* Cheap & reliable

👉 Good for reports, not for decisions

---

### Step 2: Micro-Batching (Spark)

* Instead of 1 big batch → many small batches
* Example: process every 1 second

👉 Improvement, but still not real-time

**Hidden Problem:**
Even with 1-second batch:

* Data waits in queue before processing

> In fraud detection, 1 second delay = attack success

---

### Step 3: True Streaming Systems

* Process events **as they arrive**
* No waiting
* Continuous pipelines

👉 This is where systems like:

* Storm
* Flink
* Kafka Streams
  come into picture

---

# 3. The Fundamental Reality of Streaming

Streaming is powerful… but expensive.

So we ask:

> ❓ “Should everything be real-time?”

Answer: **No.**

---

### When NOT to use streaming:

* Monthly reports
* Historical analysis
* Data warehousing

---

### When TO use streaming:

* Fraud detection
* Notifications
* Live dashboards
* Monitoring systems

---

# 4. The Core Tradeoff (The Most Important Concept)

Every streaming system is fighting this battle:

```
        Latency
        /    \
       /      \
   State ---- Scale
```

You can only optimize **2 out of 3 effectively**.

---

### Let’s connect this to real systems:

#### 1. Fraud Detection

* Needs: ultra-low latency
* Why? Delay = money loss

👉 Compromise:

* Limited state
* Moderate scale

---

#### 2. Netflix / Hotstar Analytics

* Needs: huge user session tracking

👉 Compromise:

* High state
* Slight latency acceptable

---

#### 3. IoT Systems

* Millions of sensors

👉 Compromise:

* Scale > everything else

---

#### 4. Amazon Recommendations

* Needs everything

👉 Hardest system to design

---

# 5. The Nature of Streaming Data (Why It’s Hard)

Unlike databases or files, streaming data behaves like **real life chaos**.

---

### 1. Unbounded Data

There is:

* No start
* No end

Your system:

* Never stops
* Never “completes”

---

### 2. Time Sensitivity

A late event is often useless.

Examples:

* Fraud alert after 5 mins → useless
* OTP after 30 sec → annoying

---

### 3. Unpredictable Arrival

Reality:

* Internet is messy
* Systems fail
* Packets delay

So events can:

* Arrive late
* Arrive out of order
* Come in spikes

👉 Your system must **expect imperfection**

---

# 6. Time Problem: Event Time vs Processing Time

This is where most beginners get confused.

---

### Real Scenario:

User clicks:

* At: **11:59 PM** → Event Time

System receives:

* At: **12:02 AM** → Processing Time

---

### Now problem:

You are calculating:

* “Sales per day”

Where does this belong?

---

### If you use Processing Time:

👉 Wrong results

### If you use Event Time:

👉 Correct results

---

### But here comes the twist…

What if data arrives late?

---

# 7. Watermarking (Handling Reality)

We cannot wait forever.

So we define:

> “How late is acceptable?”

---

### Example:

* Allow delay of 5 minutes
* After that → ignore or treat separately

---

### Insight:

Watermarking is basically:

> “A contract between accuracy and practicality”

---

# 8. Stateless vs Stateful — Memory vs Speed

---

## Stateless

No memory.

### Example:

* Filter logs
* Mask data

👉 Each event is independent

---

## Stateful

System remembers history.

### Example:

* Fraud detection
* Session tracking

👉 Requires:

* Storage
* Recovery
* Complexity

---

### Key Insight:

> Stateless = fast
> Stateful = powerful

---

# 9. Why Spark Falls Short

Spark was designed for **batch first**, streaming later.

---

### Core Issue:

It uses **micro-batching**

```
Collect → Wait → Process
```

---

Even if:

* Batch = 1 second

Still:

* Data waits before processing

---

### Where this fails:

* Stock trading
* Payment systems

👉 Because:

> “In real-time systems, milliseconds matter.”

---

# 10. Apache Storm — The Pioneer

Before Flink, before Kafka Streams… there was **Storm**.

---

### Big Idea:

Process **each event individually**

---

### Architecture (Think Like Pipeline):

```
Spout → Bolt → Bolt → Output
```

---

### Components:

* **Spout** → Data source
* **Bolt** → Processing logic
* **Topology** → Full pipeline
* **Nimbus** → Master
* **Supervisor** → Worker manager
* **Zookeeper** → Coordination

---

### What Storm Proved:

> Real-time distributed streaming is possible

---

### Why It Declined:

* No state management
* Complex operations
* Limited guarantees

---

# 11. Apache Flink — The Gold Standard

Flink fixed everything Storm lacked.

---

### Philosophy:

> Streaming is the primary model, batch is secondary

---

### Why Flink Wins:

* Event-by-event processing
* Exactly-once guarantees
* Native state management
* Event-time handling

---

### Architecture (Conceptual Flow):

```
Source → Stream → Operators → State → Sink
```

---

### Key Components:

* **DataStream** → Flow of events
* **Operator** → Transform logic
* **State** → Memory per key
* **Checkpointing** → Fault recovery
* **Task Manager** → Workers
* **Job Manager** → Coordinator

---

### Real Power:

* Can handle:

  * joins
  * windows
  * aggregations
  * complex logic

---

### Tradeoff:

* Expensive
* Complex setup

---

# 12. Kafka Streams — The Lightweight Approach

Not every problem needs Flink.

---

### Idea:

> “What if processing happens inside your app?”

---

### No cluster needed.

Just:

* Add a library
* Deploy like microservice

---

### Flow:

```
Kafka → App → Process → Kafka
```

---

### Core Concepts:

#### KStream

* Immutable event log

#### KTable

* Latest state view

#### State Store

* Local storage

#### Changelog

* Backup in Kafka

---

### Recovery:

* Crash → Rebuild state from Kafka

---

### Strength:

* Simple
* Cheap
* Scalable via instances

---

### Limitation:

* Limited complexity
* Java-only

---

# 13. Final Decision Framework

| Scenario                | Tool          |
| ----------------------- | ------------- |
| Simple processing       | Kafka Streams |
| Complex real-time logic | Flink         |
| Large analytics         | Spark         |

---

# Final Mental Model

Think of streaming as:

> A system that **reacts to reality as it unfolds**

---

### Final Truth:

* Batch = Understand the past
* Streaming = Control the present

---

And in modern systems:

> If you cannot react in real-time,
> you are already behind.

then next we have apache kafka stream

Apache Kafka Streams is a lightweight stream-processing library designed to process data directly inside Kafka-based applications. Unlike frameworks such as Storm or Flink, Kafka Streams does not require a separate cluster. Instead, it runs as part of a normal Java application.			
eg :
Imagine a small retail store that already receives goods through a central warehouse		
Instead of sending items to a large factory for processing, the store processes items locally—checking inventory, pricing, and discounts before putting them on shelves.		
Kafka Streams works the same way:	
Data flows through Kafka
Processing happens inside the application itself
No separate processing cluster is required.

end to end architecutre flow :
the first thing that happens is you will have a producer will write the data to kafka topic , 
then you have a kafka stream which is reading that particular topic  Kafka Streams application consumes records
Processing logic transforms or aggregates data
Local state stores maintain intermediate results
Output is written back to Kafka topics

Kafka Streams	Add a library to your Java service. Deploy it like any microservice
Kafka Topic	"The highway + warehouse. Data lives here. Multiple apps can read independently."

KStream	A live log of events. Every record is NEW. Immutable. Think: transaction history.

we have here ktable: KTable	A live snapshot. Every record UPDATES the previous. Think: current account balance.

just like check pointing we have Changelog Topic	On crash: rebuild state from this. Automatic.

Application Instance	 More instances = more parallelism.


Step 1	Application crashes. Local state store is lost.
Step 2	Kafka detects the dead instance and reassigns its partitions.
Step 3	New instance starts, reads changelog topic from Kafka.
Step 4	State store is rebuilt exactly — every key-value pair restored.

Kafka Streams — Strengths and Limitations		
		
✅ No cluster needed	Deploy as standard microservice	
✅ Kafka-native	Zero extra infra if you have Kafka	
✅ Simple to operate	Scale by adding app instances	
✅ Strong state support	Local stores + Kafka-backed recovery	
✅ Event-by-event	True low latency processing	

❌ Java/Scala only	No Python support
❌ Kafka required	Cannot run standalone
❌ Complex joins are limited	Flink handles these better
❌ Large state challenging	Local disk limits apply
