# Modern Data Architecture — Deep Dive (Connected Notes)

---

# 1. The Big Shift: How Data Architecture Evolved

Let’s start with the bigger picture.

Earlier systems were:

* Monolithic
* On-premise
* Tightly coupled (storage + compute together)

Now everything has changed.

---

## Modern Data Architecture — Core Philosophy

Today’s systems are built on:

### 1. Cloud-Native Storage

* Data lives in:

  * S3 (AWS)
  * ADLS (Azure)
  * GCS (Google)

Why?

* Cheap
* Scalable
* Durable

---

### 2. Decoupled Compute and Storage

This is one of the **most important shifts**.

Earlier:

* Storage + processing = same system

Now:

* Store data in one place (S3)
* Process using multiple engines (Spark, Flink, Trino)

---

### Why this matters:

* Same data → multiple use cases
* No duplication needed
* Flexible compute engines

---

### 3. Unified Batch + Streaming

Earlier:

* Batch systems (Hadoop)
* Streaming systems (Storm)

Now:

* Systems support both:

  * Spark (batch + micro-batch)
  * Flink (true streaming + batch)

---

### 4. Governed Access Layer

Final layer where:

* Users access data
* Policies are enforced

---

## Core Objective of Modern Architecture

> Any team should be able to access any data, at any speed, without copying it.

---

# 2. The 8 Layers of Modern Data Architecture

Think of this as a **pipeline of responsibility**, not just layers.

---

## Layer 1: Data Source Layer

This is where **data is born**.

### Sources:

* OLTP databases (transactions)
* Applications
* IoT devices
* APIs
* Logs

---

### Key Insight:

> These systems are optimized for **operations**, not analytics.

---

### Why not use directly for analytics?

* High load → system crash risk
* Not designed for heavy queries

---

## Layer 2: Ingestion Layer

This layer **moves data into the platform**.

### Tools:

* Kafka
* Pulsar

---

### Responsibilities:

* Collect data from sources
* Handle:

  * Batch ingestion
  * Streaming ingestion

---

### Important Design Point:

> Must support both real-time and historical data movement

---

## Layer 3: Storage Layer

This is the **heart of the system**.

### Tools:

* S3
* HDFS
* ADLS

---

### Data is stored in 3 forms:

---

#### Bronze Layer (Raw Data)

* Exact copy of source
* No transformation

##### Why?

* Debugging
* Reprocessing
* Source of truth

---

#### Silver Layer (Cleaned Data)

* Deduplicated
* Validated
* Enriched

##### Used by:

* Analysts
* Data scientists

---

#### Gold Layer (Curated Data)

* Aggregated
* Business-ready

##### Used by:

* Dashboards
* Executives

---

### Important Question:

Why store data 3 times?

---

### Answer (Critical Concept):

#### 1. Reliability

* If something breaks → go back to Bronze

#### 2. Flexibility

* Different teams need different formats

#### 3. Performance

* Pre-aggregated (Gold) = fast queries

---

### Example (Booking System):

* Bronze → Raw booking events
* Silver → User sessions
* Gold → Daily revenue

---

# 3. Processing Layer

This is where **data becomes useful**.

---

### Tools:

* Spark
* Flink
* MapReduce

---

### Responsibilities:

* Transform data
* Apply business logic
* Generate insights

---

### Design Pattern:

* Batch → historical data
* Streaming → real-time data

---

### Key Insight:

> Processing is where raw data becomes decision-making material

---

# 4. Analytics & Query Layer

This layer answers:

> “What is happening in the business?”

---

### Tools:

* Snowflake
* BigQuery
* Trino

---

### Characteristics:

* Read-heavy workloads
* Optimized for querying

---

### Example Queries:

* Daily revenue
* User growth
* Conversion rate

---

# 5. Machine Learning Layer

This is where systems move from:

> “What happened?” → “What will happen?”

---

### Tools:

* MLflow
* TensorFlow
* Spark MLlib

---

### Data Flow:

#### If real-time needed:

* Use streaming/structured data

#### If not real-time:

* Use data warehouse

---

### Use Cases:

* Fraud prediction
* Recommendation systems
* Demand forecasting

---

# 6. Orchestration Layer

This layer ensures everything runs **in order**.

---

### Tools:

* Airflow
* Azure Data Factory

---

### Responsibilities:

* Job scheduling
* Dependency management
* Failure handling

---

### Example:

* Run ETL after ingestion
* Run ML after processing

---

# 7. Governance Layer

This is about **trust and control**.

---

### Tools:

* Apache Atlas

---

### Responsibilities:

* Access control
* Data lineage
* Data quality

---

### Why important?

Without governance:

* Data becomes unreliable
* Security risks increase

---

# 8. Data Catalog (Implicit but Critical)

Tracks:

* What data exists
* Who owns it
* Freshness

---

# 3. Data Lake Architecture

Now let’s connect all layers into a real system.

---

## What is a Data Lake?

> A centralized storage system that stores **all data in raw form**

---

### Characteristics:

* Schema-on-read
* Stores all formats
* Cheap storage

---

### Flow:

1. Data Sources → All types
2. Ingestion → Minimal transformation
3. Storage → Raw + Processed + Curated
4. Catalog → Metadata tracking
5. Analytics → BI tools
6. ML → Uses raw data

---

### Key Insight:

> Store first, decide schema later

---

# 4. Traditional Data Warehouse Architecture

Before data lakes, we had warehouses.

---

## Characteristics:

* Centralized
* Structured data only
* Schema-on-write
* Batch ETL

---

### Flow:

1. Source systems
2. ETL process
3. Data warehouse
4. Data marts
5. BI tools

---

### Key Idea:

> Clean before storing

---

# 5. Data Lake vs Data Warehouse (Core Difference)

| Feature     | Data Lake      | Data Warehouse |
| ----------- | -------------- | -------------- |
| Schema      | On-read        | On-write       |
| Data Type   | Any            | Structured     |
| Flexibility | High           | Low            |
| Cost        | Low            | High           |
| Speed       | Slower queries | Faster queries |

---

### When to Use Warehouse:

* Structured data
* Stable schema
* Compliance-heavy systems

Examples:

* Finance
* Healthcare

---

# 6. Dimensional Modeling (Warehouse Design)

To make warehouses efficient, we use:

---

## Fact Table

* Contains:

  * Metrics (measurable values)

Examples:

* Sales amount
* Quantity

---

## Dimension Table

* Contains:

  * Descriptive attributes

Examples:

* User details
* Product info

---

## Star Schema

```
       Dim
        |
Dim — Fact — Dim
        |
       Dim
```

* Simple
* Fast queries

---

## Snowflake Schema

```
Fact → Dim → Dim
```

* More normalized
* Less redundancy
* More joins

---

# Final Mental Model

Modern architecture is about:

> Decoupling everything, scaling independently, and enabling real-time + batch together

---

### Final Connection Across Concepts:

* Streaming → real-time decisions
* Storage layers → reliability + flexibility
* Processing → transformation
* Warehouse → structured analytics
* ML → predictions
* Governance → trust

---

### Final Truth:

> The goal is not just to store data
> but to make it **usable, reliable, and actionable at any moment**
